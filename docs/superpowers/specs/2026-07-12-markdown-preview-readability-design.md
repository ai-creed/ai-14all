# Markdown Preview — Readability Across Themes — Design

**Date:** 2026-07-12
**Status:** Approved design, pending implementation plan
**Scope:** Full readability treatment for the markdown preview surface (`MarkdownBody` and every surface that renders markdown through the shared `.shell-md-modal__body`/`.shell-md-preview__body` styles: the inline `MarkdownPreview`, `MarkdownPreviewModal`, and the session-note preview in `NoteSheet`): restore what Tailwind preflight stripped (heading scale, list markers, `hr`), GitHub-README typography with a dedicated reading font, and token-driven per-theme syntax highlighting replacing the hardwired `github-dark.css`. CSS + one component wrapper + tokens + one new font dependency; no renderer/plugin changes.

## 1. Context

Follow-up to the review-pane UX hardening pass (2026-07-11), which introduced the shared `MarkdownBody` renderer and made preview the default for `.md` files. User feedback: rendered markdown "reads badly in all themes." A code audit confirmed the complaints and found the root causes.

The app has four themes, all defined as token blocks in `src/styles/tokens.css`: dark (`:root`, the default), `[data-theme="light"]`, `[data-theme="warm"]` (dark warm-brown surface), and `[data-theme="tui"]` (WebTUI-style stepped ladder, `docs/tui-css-spec.md`). Tailwind v4 is imported at the top of `tokens.css` (`@import "tailwindcss"`, line 1), so **preflight resets are active globally** — and the markdown styles in `shell.css` only partially restore them.

| # | Problem (evidence) |
|---|--------------------|
| P1 | **Headings render at body-text size in every theme.** Preflight resets h1–h6 to `font-size: inherit`; `shell.css:2661-2677` restores only `font-weight: 600` and margins, never `font-size`. Documents have no visual hierarchy. |
| P2 | **List markers are invisible in dark, light, and warm.** Preflight sets `list-style: none` on ul/ol; `shell.css:2729-2735` restores `padding-left` only. Only the TUI theme patched markers back (`tui.css:394-403` — its own comment concedes "base leaves them list-style:none"). |
| P3 | **Code blocks hardwire the github-dark palette.** `MarkdownBody.tsx:4` imports `highlight.js/styles/github-dark.css` unconditionally. The light theme is re-colored by an override file (`src/styles/hljs-light.css`, scoped `[data-theme="light"]`); warm gets a cold blue-black `#0d1117` box inside its warm-brown panel; TUI ignores its ladder spec. Additionally the stock stylesheet gives `.hljs` its own background and padding *inside* the already-padded themed `<pre>` (`shell.css:2684-2691`), producing a nested double box in every theme. |
| P4 | **`hr` is near-invisible** — preflight collapses it and `shell.css` has no rule for it. |
| P5 | **GFM task-list checkboxes are unstyled** and will collide with restored list markers (double marker) unless task-list items opt out of `list-style`. |
| P6 | **Prose is set in monospace.** The app-wide `--font-ui` is itself a mono stack (`shell.css:60-61`, "SF Mono" …), so preview body text inherits monospace — the direct source of the "hard to read" feedback. |
| P7 | **No reading measure.** The inline preview body spans the full pane width; long lines hurt readability. |
| P8 | **Structural smell:** every markdown rule is duplicated as `.shell-md-modal__body …, .shell-md-preview__body …` selector pairs (~12 pairs in `shell.css:2654-2754`, plus `:is()` pairs in `tui.css`) because `MarkdownBody` has no root class of its own. Worse, a third surface piggybacks on those class names: `NoteSheet` renders its own `ReactMarkdown` (same remark-gfm + rehype-highlight stack) directly inside `<div className="shell-md-modal__body shell-note-sheet__preview-body">` (`src/features/workspace/components/NoteSheet.tsx:89-95`), so the "modal body" styles are silently load-bearing for the session-note preview. |

Already solid — not touched by this pass: link clicks are guarded by `electron/main/services/navigation-guard.ts` (external links open in the OS browser; `file:`/`javascript:`/`data:` blocked); react-markdown leaves raw HTML unrendered (default, safe); `MarkdownPreviewModal`'s always-mounted Radix workaround; the `files:read` IPC path and its failure surface.

## 2. Decisions

### Architecture & file layout

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | `MarkdownBody` wraps its output in a root `<div className="shell-md-body">`; all duplicated `.shell-md-modal__body …, .shell-md-preview__body …` selector pairs in `shell.css` and the `:is()` pairs in `tui.css` collapse to `.shell-md-body …`. The wrapper classes survive for layout only (padding, scroll). **`NoteSheet` migrates to the shared renderer**: its inline `ReactMarkdown` block (`NoteSheet.tsx:89-95`, plugin stack identical to `MarkdownBody`'s) is replaced by `<MarkdownBody content={note} />`, dropping the now-unneeded `shell-md-modal__body` class; `shell-note-sheet__preview-body` stays for note-specific layout. The note preview thereby inherits the full readability treatment (typography, reading font, themed code blocks) instead of losing its styling to the collapse. | Single-point document styling; future theme fixes touch one selector. Fixes P8 including its NoteSheet half — without the migration, collapsing the `.shell-md-modal__body` rules would strip the session-note preview's markdown styling. |
| D2 | Delete the `github-dark.css` import from `MarkdownBody.tsx` and delete `src/styles/hljs-light.css`. Replace both with one new `src/styles/hljs-tokens.css` mapping highlight.js scope classes to `var(--hljs-*)` custom properties; per-theme values live in `tokens.css` beside every other token. The `main.tsx` import of `hljs-light.css` is replaced by `hljs-tokens.css`. | Code blocks become themed the same way everything else is. Fixes P3's palette half. |
| D3 | `tui.css` sheds patches that become base behavior: the ul/ol `list-style` restore (`tui.css:394-403`) is deleted once D8 lands it in base. The TUI inline-code/link `--accent` → `--primary` pin **stays** (TUI's `--accent` resolves to a dark ladder step — a token-bridging quirk, not a markdown bug), re-targeted to `.shell-md-body`. | Base owns correctness for all themes; TUI keeps only genuinely TUI-specific overrides. |
| D4 | No renderer/plugin changes: react-markdown + remark-gfm + rehype-highlight stay as-is; raw HTML stays unrendered. | This pass is CSS + one component wrapper + tokens + a font package. |

### Typography (GitHub-README treatment)

All rules on `.shell-md-body`; colors exclusively from existing theme tokens so every theme inherits correctly.

| # | Decision | Rationale |
|---|----------|-----------|
| D5 | ~~Reading measure: `.shell-md-body { max-width: 72ch; margin-inline: auto; }`~~ **REVISED 2026-07-13 (Vu, post-landing):** no width cap — `.shell-md-body` fills whatever width its pane provides. The original 72ch measure shipped, was reviewed in the app, and was reverted by operator decision: wasted space on wide panes outweighed long-line fatigue. P7 is accepted as-is. | Operator preference after seeing it live: use the available space. |
| D6 | Heading scale: h1 `2em`, h2 `1.5em`, h3 `1.25em`, h4 `1em`, h5 `0.875em`, h6 `0.85em` + `color: var(--text-muted)`. h1 and h2 get a bottom rule (`1px solid var(--panel-border)`, `padding-bottom: 0.3em`). All headings: `line-height: 1.25`, `margin-top: 1.5em`, `margin-bottom: 0.5em`, weight 600 (kept). Em-based margins replace the current `--space-*` heading margins. | Restores the hierarchy preflight stripped (P1); em-based spacing scales with the heading, GitHub-style. |
| D7 | Margin trim: `.shell-md-body > :first-child { margin-top: 0 }`, `> :last-child { margin-bottom: 0 }`. | Kills the stray gap when a document starts with a heading. |
| D8 | Lists: `ul { list-style: disc }`, `ol { list-style: decimal }`, `ul ul { list-style: circle }`, `ul ul ul { list-style: square }`, `li + li { margin-top: 0.25em }`, `li > ul, li > ol { margin-bottom: 0 }`. | Restores markers in all themes (P2); nested-marker progression mirrors browser defaults preflight erased. |
| D9 | Task lists: `li.task-list-item { list-style: none }` with negative-indent alignment so checkboxes sit where bullets would; checkboxes get `accent-color: var(--accent)`, a right margin, and vertical centering. | GFM checkboxes read as checkboxes, no double markers (P5). |
| D10 | `hr { border: 0; border-top: 2px solid var(--panel-border); margin: 1.5em 0 }`. | Fixes P4. |
| D11 | Tables: keep current border/`th` treatment; add zebra striping `tbody tr:nth-child(2n) { background: var(--panel-bg-elevated) }`; tables get `display: block; overflow-x: auto` so a wide table scrolls instead of blowing out the pane. | GitHub-familiar; keeps wide tables from forcing horizontal page scroll (originally guarded the D5 measure; still wanted without it). |
| D12 | Images: `max-width: 100%; border-radius: var(--radius-sm)`. | Contained and consistent with app surfaces. |
| D13 | Body rhythm: `line-height: 1.6` (kept); block elements (`p`, `pre`, lists, tables, blockquote) normalize to `margin-bottom: 0.75em`, replacing the mixed `--space-3` values. Blockquote, links, `del`, and inline-code treatments are unchanged apart from the selector collapse. | Visually near-identical, internally consistent; em-based rhythm follows the reading font size. |
| D18 | Reading font: new dependency `@fontsource-variable/hanken-grotesk` (self-hosted, imported in `main.tsx` — no network fetch, works packaged). New token `--font-reading: "Hanken Grotesk Variable", system-ui, -apple-system, "Segoe UI", Roboto, sans-serif;` in the `shell.css` root block. `.shell-md-body { font-family: var(--font-reading); font-size: 14px; }` — one point above the app's 13px body, since this is a reading surface. Headings use the same family (weight 600 per D6). Code blocks and inline code explicitly stay `var(--font-terminal)`. Applies in **all four themes**, TUI included (readability wins on a document surface; decided explicitly). Everything outside `.shell-md-body` keeps the mono chrome. | The app-wide `--font-ui` is monospace (P6); Hanken Grotesk is the proven body voice of ai-samantha's onboarding wizard (its `--font-ui`), a warm humanist sans with sane system fallbacks. |

### Code blocks (token-driven highlighting)

| # | Decision | Rationale |
|---|----------|-----------|
| D14 | Single box: `.shell-md-body pre` keeps owning the visual box (`background: var(--panel-bg-elevated)`, padding, radius, `overflow-x: auto`); new rule `.shell-md-body pre code.hljs { background: transparent; padding: 0 }` neutralizes the vendored box. | No more `#0d1117` island inside a themed panel (P3's box half). |
| D15 | `hljs-tokens.css` maps scopes onto 13 semantic vars, reproducing the GitHub palettes' grouping **exactly** (verified against `node_modules/highlight.js/styles/github-dark.css` and `src/styles/hljs-light.css`, which share the same group structure): `--hljs-fg` (base text, `subst`), `--hljs-comment` (`comment`, `code`, `formula`), `--hljs-keyword` (`keyword`, `doctag`, `meta .keyword`, `template-tag`, `template-variable`, `type`, `variable.language_`), `--hljs-entity` (`title`, `title.class_`, `title.class_.inherited__`, `title.function_`), `--hljs-constant` (`attr`, `attribute`, `literal`, `meta`, `number`, `operator`, `variable`, `selector-attr`, `selector-class`, `selector-id`), `--hljs-string` (`string`, `regexp`, `meta .string`), `--hljs-builtin` (`built_in`, `symbol`), `--hljs-name` (`name`, `quote`, `selector-tag`, `selector-pseudo`), `--hljs-section` (`section`, bold — the only non-color trait either palette attaches to a color group), `--hljs-bullet` (`bullet`, color only), `--hljs-addition-fg`/`--hljs-addition-bg`, `--hljs-deletion-fg`/`--hljs-deletion-bg`. `emphasis` → italic and `strong` → bold carry no color var. The scopes GitHub's palettes purposely leave unstyled (`char.escape_`, `link`, `params`, `property`, `punctuation`, `tag`) stay unstyled and inherit the base color — `hljs-tokens.css` must NOT invent rules for them. Unlanguaged code blocks get no hljs classes → plain base color on the themed box. | One var per GitHub color group — no group is merged or split, so D16's carry-over of the dark and light palettes (starting values, contrast-gated per D16/D17) is expressible exactly. `built_in`/`symbol` (own orange) and `bullet` (own gold) get dedicated vars; a merged mapping could not reproduce the existing palettes. |
| D16 | Per-theme values in `tokens.css`, with **D17's contrast gate as the binding constraint and the inherited palettes as starting values, not verbatim requirements**: **dark** starts from github-dark's colors and **light** from the `hljs-light.css` values (then the file dies); any starting value that fails the D17 gate is minimally adjusted — same hue, lightness nudged just far enough to clear 4.5:1 — and the adjustment is commented in `tokens.css` with the before/after ratio, following the precedent of the AAA token work. Known offenders to adjust (measured against each theme's `--popover` code background): light `--hljs-builtin` `#e36209` ≈ 3.49:1 on white (`tokens.css:81`), dark `--hljs-section` `#1f6feb` ≈ 3.9:1 on `oklch(0.208 0.042 265.755)` (`tokens.css:21`), and the light addition pair `#22863a` on `#f0fff4` ≈ 4.48:1 — the implementer re-measures all values and adjusts whatever the T1 test flags, not just these three. **warm** gets a new warm-tinted palette (amber keyword family near `--primary #e58a5e`, sand strings, khaki comments) and **tui** a ladder-conformant palette per `docs/tui-css-spec.md` (cursor-teal `--primary` for keywords, xterm-style accents) — exact hexes picked at implementation time, same gate. | Palette-native code blocks per theme; a future theme just defines vars. Verbatim carry-over and the contrast gate proved mutually incompatible (three current GitHub values fail 4.5:1 on the app's backgrounds), and the gate is the point of this pass — so the gate wins and palettes are near-identical starting points. |
| D17 | Contrast gate as a unit test: parse `tokens.css`, resolve each theme's `--hljs-*` foreground values against that theme's code-block background (`--panel-bg-elevated` → `var(--popover)`, `tokens.css:39` — so `--popover` per theme), and assert WCAG contrast **≥ 4.5:1** for every foreground var per theme; `--hljs-addition-fg`/`--hljs-deletion-fg` are checked against their own bg vars. | Promotes the audit wish in `theme-wcag-aa-spec.md` §6 to CI, scoped to the new tokens. Palette tweaks that regress readability fail the build. |

## 3. Testing

| # | Test | Notes |
|---|------|-------|
| T1 | `tests/unit/styles/hljs-contrast.test.ts` — the D17 contrast gate. | Pure text parsing + color math (oklch/hex → relative luminance → ratio); no DOM. |
| T2 | `MarkdownBody` unit tests (extend the existing viewer suite): root `shell-md-body` class present; GFM active (table + task-list checkbox render); fenced code gets hljs classes. Plus NoteSheet migration coverage: the note preview renders through `MarkdownBody` (a `.shell-md-body` root appears inside `.shell-note-sheet__preview-body`, and no `shell-md-modal__body` class remains in `NoteSheet`). | Guards D1 — including the third consumer whose styling the selector collapse would otherwise silently strip — and the renderer contract. |
| T3 | Preflight-restoration pin: parse `shell.css` and assert `.shell-md-body` rules define `font-size` for h1–h6, `list-style` for ul/ol, and an `hr` border. | jsdom doesn't apply real stylesheets, so text-level assertion is the honest unit-level tool. Fails loudly if someone "cleans up" the restorations without knowing about preflight. |
| T4 | Import hygiene: assert `MarkdownBody.tsx` no longer imports `highlight.js/styles/*`, and `hljs-light.css` is gone from `main.tsx` and from disk. | Prevents the vendored theme sneaking back. |
| T5 | e2e smoke (extend the existing viewer-preview spec): open a fixture `.md` in preview; assert computed `font-family` contains "Hanken Grotesk Variable", h1 computed size > body size, `ul` has `list-style-type: disc`, code-block background ≠ `rgb(13, 17, 23)`. One extra scenario: switch dark → light and assert code-block token colors change. | Proves the token indirection end to end in a real browser. |
| T6 | Release gate unchanged: typecheck, lint, prettier, full unit, full e2e (`CI=1 ZDOTDIR=$(mktemp -d) pnpm test:e2e`, real exit code) before commit. | Same bar as the previous pass. |

Visual verification: run the dev app and eyeball a real spec document in all four themes before sign-off — font and palette judgments need eyes, not just assertions.

## 4. Out of scope

- Fraunces (ai-samantha's display serif) — samantha uses it only for headline moments; this pass keeps one reading family.
- Monaco/diff-editor theming, app-chrome typography, and any surface outside `.shell-md-body`.
- Markdown feature additions (raw HTML, footnotes plugins, mermaid, math) — renderer stack is frozen (D4).
- Link-click behavior — already guarded by `navigation-guard.ts`.
- The review-pane deferred backlog (A5–A7, A11–A16) — still parked in the session note.
