# ai-14all — Structural Redesign Spec

Spec for the session-elevation + terminal-hero redesign. This document describes the desired product behavior and acceptance criteria. It is not an implementation plan.

## 0. How to use this document

Read these artifacts together:

- **`UI-POLISH.md`** — detail-level polish items and bug fixes. Most of it can land independently of this redesign.
- **`REDESIGN.md`** *(this file)* — structural product direction: make sessions the visible unit of work, make the terminal the main surface, and move review/files/notes into summoned or compact surfaces.
- **`ui-polish-prototype.html`** — interactive visual reference with Before / Polish / Redesign modes. Treat it as a visual aid, not a pixel-perfect implementation contract.

This spec supersedes these `UI-POLISH.md` items:

- `§3.1 Session Info collapsed strip` → replaced by the session chip bar.
- `§3.3 Review stack auto-collapse when clean` → replaced by the review drawer.
- `§3.2 Session Note placeholder` → replaced by the note sheet.
- `§6.1 WorktreeTree header toolbar` → incorporated into the Files overlay.

Everything else in `UI-POLISH.md` remains applicable unless it conflicts with this spec.

## 1. Product intent

The redesign is driven by three commitments.

**The terminal is the hero.** The main session column should default to a terminal-dominant layout. Session metadata, notes, files, and Git review should support the terminal workflow without permanently competing for equal vertical space.

**Sessions are the user's mental model; worktrees are the implementation.** The app should present the active unit of work as a session with a human-readable title. Worktree folder name, branch, and path remain visible as supporting metadata, not the primary label.

**Periphery is summoned.** File browsing, session notes, and deep review are used intermittently. They should be quickly available through explicit controls and shortcuts, but they should not occupy permanent full panels by default.

## 2. Scope

In scope:

- Session title as the primary label in the rail and main session chrome.
- Optional user rename for every worktree-backed session, including the main worktree session.
- Thin session chip bar replacing the expanded top band.
- Collapsible review drawer replacing the always-expanded review stack.
- Files overlay as the primary file-browsing entry point.
- Note sheet as the primary note surface.
- Keyboard shortcuts and focus behavior for the new summoned surfaces.
- Dark and light theme parity for every new visible surface.
- Focus management and keyboard-only reachability for every new interactive surface.

Out of scope:

- Multiple sessions sharing one worktree.
- Multi-worktree comparison.
- Editable embedded code as a primary workflow. The existing narrow lightweight editor remains available for whitelisted files; this redesign must not broaden that editor's scope.
- Remote/collaboration features.
- Command palette (`Cmd+K`) beyond a basic shortcuts/help overlay.
- Advanced Git client behavior beyond lightweight review.

## 3. Product contracts

### 3.1 Session identity

For this redesign, a session remains 1:1 with a worktree.

`WorktreeSession.title` represents an optional user-defined session name. The semantic contract is:

- Empty stored title means "no custom title."
- Display title is `session.title.trim() || worktree.label`.
- A non-empty stored title means the user intentionally named the session.
- Clearing the rename input stores an empty title and returns the UI to the worktree-label fallback.
- New and restored sessions that have no explicit custom name must store an empty title, not a copy of `worktree.label`.

This distinction is required so the app can tell a default worktree label apart from a user-authored session name.

### 3.2 Session-scoped state

The following state belongs to the active worktree session:

- Session title.
- Session note.
- Selected file/change/commit review context.
- Review drawer expanded/collapsed state.
- Terminal layout state already scoped to the worktree session.

The following may remain transient view state unless future usage proves persistence is needed:

- Current Files overlay open/closed state.
- Current Note sheet open/closed state.
- Review drawer user-collapse suppression during the current app runtime.
- Review drawer dimensions and review rail dimensions.

Any rail or chip-bar view model must include session-scoped data, not just worktree data. In particular, those surfaces need access to the display title, note presence, dirty count, branch metadata, and process attention for each visible session.

### 3.3 Security and architecture boundaries

The renderer remains unprivileged. New file, Git, persistence, and terminal behavior must continue to go through preload and typed IPC contracts. Product logic that outlives a component render should live in services or shared state modules, not be embedded ad hoc in React components.

The redesign should reduce the weight of `App.tsx` over time, but this spec does not require a particular file split. Implementation plans may choose the safest extraction sequence.

## 4. Required surfaces

### 4.1 Session rail

The rail should make named sessions easy to scan.

Each expanded worktree card should show:

1. Session title as the primary label.
2. Worktree label as subordinate metadata when the session has a custom title.
3. Branch name as subordinate metadata.
4. Dirty count when the session has changed files.
5. Process summary and attention state.

When no custom title exists, the card should collapse title and worktree label into the current single primary label. The UI should not show redundant title and worktree lines with identical text.

Rename behavior:

- Double-clicking the session title starts inline rename.
- A context-menu item starts rename.
- The chip-bar rename action starts the same rename flow.
- If the rail is collapsed, a rename action expands the rail and focuses the inline rename field.
- `Enter` commits, `Esc` cancels, and blur commits the current trimmed value.
- Empty or whitespace-only input clears the custom title.

The footer action should say `+ New session`. Creation may still create a Git worktree under the hood, but the user-facing noun should be session. The creation flow should allow an optional session title and should still make branch/worktree naming clear.

### 4.2 Session chip bar

The top band is replaced by a compact chip bar that keeps the active context visible without consuming a large panel.

It should contain, from left to right:

```text
[Session title] [rename] · [worktree · branch] · [state]     [Files] [Note]
```

Required behavior:

- Session title uses the display-title policy from §3.1.
- Rename action is visible on hover and keyboard-focusable.
- Worktree and branch are always available as metadata.
- Clean state is visible but visually quiet.
- Dirty state shows the changed-file count and opens/expands review on activation.
- Files opens the Files overlay.
- Note opens the Note sheet and shows a small indicator when the session note is non-empty.

The chip bar replaces the old expanded session-header/note top band. Session notes move to the Note sheet; session path and Git state remain discoverable through chip-bar metadata and review surfaces.

### 4.3 Review drawer

The review surface becomes a drawer anchored at the bottom of the main session column.

Collapsed state:

- Always visible when a worktree session is active.
- Approximately one compact row tall.
- Shows a chevron, `Review`, clean/dirty state, changed-file count, and refresh action.
- Refresh must not toggle drawer expansion.

Expanded state:

- Opens upward in place so the terminal remains visible above it.
- Hosts the existing review modes: Files, Changes, and Commits.
- Keeps file viewing and diff/commit inspection inside the session workflow.
- May remain resizable.

Files tab policy:

- The Files overlay is the primary file-browsing entry point.
- The drawer should keep the Files mode during this redesign so users still have a persistent file-view option and existing review/viewer behavior does not disappear.
- Removing the drawer's Files mode is a separate future product decision.

Auto-expand policy:

- If a session transitions from zero changed files to one or more changed files during the current app runtime, the drawer should expand unless the user explicitly collapsed it for that session during the same runtime.
- Loading an already-dirty session from persistence should not be treated as a new transition.
- Explicit user collapse should suppress future auto-expand for that session until the session is switched away and back, the app restarts, or the user expands it manually.
- New sessions default collapsed; restored sessions honor their persisted drawer state.
- The persisted expanded/collapsed value should reflect the user's last stable preference, not a transient animation.

### 4.4 Files overlay

The Files overlay is the primary way to browse files without permanently occupying the review rail.

Shape:

- Centered modal sized for fast keyboard and pointer navigation.
- Uses the app's modal/focus-trap conventions.
- Has a search input at the top.
- Shows a virtualized file tree/list with Git status indicators.
- Has a footer with current selection and available actions.

Behavior:

- Opening the overlay focuses search.
- Search is fuzzy or prefix-tolerant enough to find nested files quickly.
- Arrow keys move selection.
- `Enter` views the selected file, closes the overlay, selects the file in the session, and ensures the drawer can show the viewer.
- `Cmd/Ctrl+Enter` opens the lightweight editor only when that file is editable under the app's existing editor whitelist; otherwise the action is disabled or a no-op with clear feedback.
- `Esc` closes without changing selection.
- Selecting a file through pointer interaction follows the same view behavior as `Enter`.

The overlay may reuse existing tree code or extract a shared tree model. The spec requires the behavior, not a particular component reuse strategy.

### 4.5 Note sheet

The session note moves from permanent top-band chrome into a right-side sheet.

Shape:

- Right-side sheet, full height, sized for short session notes.
- Opens from the chip bar Note action and from the note shortcut.
- Contains a textarea with a clear placeholder.
- Uses the app's modal/sheet focus behavior.

Behavior:

- Opening the sheet focuses the note textarea.
- Note contents persist per worktree session.
- Closing the sheet never discards typed content.
- The chip bar shows a note indicator whenever `session.note.trim()` is non-empty.

### 4.6 Terminal section

The terminal section remains the primary work surface. The redesign does not change terminal semantics; it gives the terminal more default space by shrinking surrounding chrome.

Polish items from `UI-POLISH.md` still apply:

- Hover/focus-revealed terminal tab close action.
- Clear split-mode active state.
- Shorter empty split-pane copy.

## 5. Keyboard and focus behavior

Keyboard shortcuts must be designed around a terminal-first app. Existing terminal behavior is part of the product's core functionality and must keep working after this redesign.

Shortcut ownership:

- Focused terminal: terminal-local shortcuts and xterm behavior win.
- Focused non-terminal shell UI: app-level shortcuts may run.
- Focused text input, textarea, Monaco surface, or dialog: that surface owns typing and its documented local shortcuts.
- App-level handlers must not use capture-phase interception to override terminal-owned bindings.
- Capture-phase handling is allowed only for explicitly reserved app accelerators that are proven not to break terminal behavior and are listed in the shortcut registry.

Required shortcut model:

| Action | macOS | Windows/Linux | Notes |
|---|---|---|---|
| Open Files overlay | `Cmd+P` | `Ctrl+Shift+P` | Runs from non-terminal focus. If the terminal owns the focused key event, terminal behavior wins. |
| Open Note sheet | `Cmd+;` | `Ctrl+;` | Runs from non-terminal focus. Prefer this over `Cmd/Ctrl+N` to avoid new-window/new-line conflicts. |
| Toggle Review drawer | `Cmd+J` | `Ctrl+J` | Runs from non-terminal focus. Must not intercept terminal newline behavior or existing reload shortcuts. |
| Rename current session | `Cmd+Shift+R` | `Ctrl+Alt+R` | Runs from non-terminal focus. Must also be available through chip-bar and rail controls. |
| Show shortcuts | `Cmd+?` or `Cmd+/` | `Ctrl+?` or `Ctrl+/` | Runs from non-terminal focus unless verified safe for terminal focus; must list platform-specific shortcuts. |

Shortcut supersession:

- This redesign supersedes the `UI-POLISH.md §7.3` `Cmd/Ctrl+Shift+R` refresh-review shortcut where it conflicts with the platform-specific rename binding.
- Refresh remains available from visible refresh controls in the review drawer and file surfaces.
- Any remaining `UI-POLISH.md §7.3` shortcuts should be revalidated against the terminal-first ownership rules before implementation.

Focus rules:

- Global app shortcuts should use a single shortcut registry or equivalent source of truth so handlers and help text cannot drift.
- Text inputs, textareas, modal editors, and open dialogs own ordinary typing. App-level shortcuts should only intercept the specific modified combinations listed above.
- While a modal/sheet is open, its local shortcuts take priority.
- Every mouse-only action introduced by this redesign must have a keyboard path.

## 6. Theme, accessibility, and visual requirements

The prototype communicates density, hierarchy, and approximate layout. The implemented UI must follow the app's existing design tokens rather than copying hardcoded prototype styles blindly.

Requirements:

- New surfaces must work in dark and light themes in the same change that introduces them.
- Text must remain readable and non-overlapping at narrow desktop widths.
- Controls must expose accessible names.
- Drawer expansion/collapse state should be announced or otherwise understandable to assistive tech.
- Overlays and sheets must trap focus while open and restore focus to the triggering control when closed.
- Reduced-motion users should not be forced through large animated movement.

## 7. Rollout slices and acceptance criteria

These slices describe independently reviewable product outcomes. They are not a file-by-file implementation plan.

### Slice A — Session title contract

Acceptance criteria:

- Session title persistence supports the empty-title fallback contract from §3.1.
- Old persisted workspaces without titles load without visual change.
- New sessions with no custom title store an empty title.
- Clearing a custom title returns display to the worktree label.
- Unit coverage exists for title set, title clear, and old-snapshot hydration.

### Slice B — Session-elevated rail

Acceptance criteria:

- The rail displays session title first when a custom title exists.
- Worktree label and branch are subordinate metadata.
- Rename works by double-click, context menu, and keyboard-accessible command.
- Collapsed rail rename has a defined, working behavior.
- Dirty count and process attention remain visible enough for fast scanning.

### Slice C — Chip bar

Acceptance criteria:

- The old expanded top band no longer occupies permanent vertical space.
- The chip bar exposes session title, rename, worktree, branch, clean/dirty state, Files, and Note.
- Dirty state opens/expands review.
- Note indicator reflects non-empty note content.
- Dark/light theme parity is covered.

### Slice D — Review drawer

Acceptance criteria:

- New-session review is compact by default and expandable on demand; restored sessions honor persisted drawer state.
- Existing Files, Changes, and Commits workflows remain available when expanded.
- Refresh does not toggle expansion.
- Auto-expand only fires on a real clean-to-dirty transition in the current runtime.
- User collapse suppression behaves per §4.3.
- E2E coverage covers collapsed, expanded, dirty transition, and persistence behavior.

### Slice E — Files overlay

Acceptance criteria:

- Files opens from chip bar and shortcut.
- Search, keyboard selection, view, editable-file edit, and close behavior match §4.4.
- File selection closes the overlay and routes to the session viewer.
- Focus is trapped and restored correctly.
- Existing file tree/review coverage is extended rather than replaced.

### Slice F — Note sheet

Acceptance criteria:

- Note opens from chip bar and shortcut.
- Text persists per session.
- Chip-bar note indicator updates as content becomes empty/non-empty.
- Focus and close behavior match §4.5.

### Slice G — Shortcut help

Acceptance criteria:

- Shortcut help lists the platform-specific bindings from §5.
- The same shortcut definitions drive handlers and help text, or tests prevent drift.
- Existing terminal-owned shortcuts continue to work when the terminal is focused.
- Help is accessible from keyboard.

## 8. Open product decisions

These should be resolved before implementation planning if they matter to the next slice.

- Whether drawer dimensions should become per-session persisted preferences. Default answer for this redesign: no.
- Whether the Files mode should eventually be removed from the review drawer. Default answer for this redesign: keep it.
- Whether `+ New session` should expose both worktree name and session title on the first release of this redesign. Default answer: yes, because the distinction is central to the new mental model.

## 9. Verification expectations

Every shipped slice should include:

- Targeted unit tests for state and persistence changes.
- E2E coverage for new user-visible behavior.
- Manual smoke pass on a full session: create session, rename it, create a dirty change, open Files, inspect review, edit an allowed file, write a note, switch sessions, restart/restore.
- Visual comparison against the prototype for hierarchy and density, not pixel-perfect matching.
- Dark and light theme check.
- Keyboard-only pass for all new interactions.

New user-visible behavior in this redesign is not complete until e2e coverage covers it.
