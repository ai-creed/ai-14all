# ai-14all — Pre-Beta UI Polish Plan

A ranked, concrete polish pass for the core surfaces (sessions rail, main column) and secondary UI (dialogs, menus, viewer, editor) before inviting real mainstream-developer testers.

Audience target informing this doc: **mainstream developers** — keep the terminal-first aesthetic, but add affordances, hover feedback, and empty-state guidance so users who aren't the original author can find everything on their own.

Scope explicitly *not* covered here (by choice during intake): design tokens/motion/theme parity, and dedicated copy/keyboard/accessibility passes. Items in those areas show up where they directly block the core/secondary surfaces, but a deeper system-level pass is deferred.

---

## TL;DR — top 10 highest-impact fixes

Ranked by impact × cost-to-fix. The first four are real bugs or near-bugs; the rest are polish items that will most change how the app *feels* on first open.

1. **Fix broken `shell-btn` styles in three dialogs.** `ConfirmCloseDialog`, `SaveConflictDialog`, and `EditorModal`'s Save button all use `shell-btn` / `shell-btn--primary` / `shell-btn--danger` — but no matching CSS rule exists in `shell.css`. These buttons currently render with browser defaults. First-time beta testers will hit this the moment they edit a file. See §5.1.
2. **Rebuild `PresetManager` to match the rest of the app.** The dialog uses inline `style={{...}}` props throughout, native buttons with no `shell-button` class, and has an unconfirmed `Delete` action. It looks like a different app. See §5.5.
3. **Normalize the dialog component system.** `shell-modal` vs `shell-md-modal` vs `shell-editor-modal` vs `shell-preset-dialog` are four near-duplicate component implementations. Collapse to one `Modal` primitive with size variants. See §5.6.
4. **Stop the review stack from eating the terminal when clean.** When there are 0 changed files the review stack still occupies `reviewPanelHeight`px and the header + rail + empty-state viewer burn 240+ px. Auto-collapse when there's nothing to show. See §3.3.
5. **Add a "dirty" indicator to the editor modal title** (`• relativePath` while dirty). Currently no visual signal other than the Save button's disabled state. See §5.2.
6. **Strengthen active-workspace signaling in the rail** with a colored left-bar using `--pane-border-sessions`. The current blue border tint is easy to miss at a glance. See §2.1.
7. **Session Info: collapse to one line once a session is loaded.** The expanded header duplicates path/branch/status info and steals vertical real estate from the primary work surface. See §3.1.
8. **Give `WorktreeTree` a visible refresh button** (currently only right-click on the root row). Mainstream devs won't discover that. See §6.1.
9. **Standardize the expand/collapse and close glyph vocabulary** — currently a mix of `▸◂`, `▴▾`, `×`, `✕`, and text "Close". See §7.1.
10. **Keyboard shortcut map.** Only `Cmd+E` (open editor) and `Cmd+S` (save, inside editor) exist globally. Ship with a minimum of `Cmd+B` (toggle rail), `Cmd+T` (new shell), `Cmd+W` (close active shell), `Cmd+1..9` (switch shells), `Cmd+\` (split), `Cmd+?` (show shortcut help). See §7.3.

## 2. Sessions rail (`SessionSidebar.tsx` + `.shell-sidebar__*`)

### 2.1 Strengthen active-workspace and active-worktree signaling — 🔴 high impact

**Problem.** `data-active-workspace="true"` only tints the group border to `rgba(98, 180, 255, 0.35)` with a 0.08-alpha background. At a glance with four workspaces stacked, identifying the active one takes a beat. The active worktree row (`data-selected="true"`) uses a 1 px `--accent` border + elevated background — slightly stronger but still easy to confuse with `--panel-bg-elevated` hover states.

**Proposal.**

```css
/* shell.css */
.shell-sidebar__workspace-group[data-active-workspace="true"] {
  position: relative;
  border-color: rgba(98, 180, 255, 0.35);
  background: rgba(98, 180, 255, 0.08);
}
.shell-sidebar__workspace-group[data-active-workspace="true"]::before {
  content: "";
  position: absolute;
  left: -1px; top: 8px; bottom: 8px;
  width: 2px;
  background: var(--pane-border-sessions);
  border-radius: 2px;
}

.shell-sidebar__item[data-selected="true"] {
  background: var(--panel-bg-elevated);
  border-color: var(--accent);
  box-shadow: inset 2px 0 0 var(--accent);
}
```

Result: a 2 px left bar on the active workspace group *and* the active worktree row — two levels of signaling without extra chrome. Passive workspaces should lose their hover tint so only the active group looks "live."

### 2.2 Differentiate workspace and worktree hierarchy — 🟡 medium

**Problem.** Workspace headers (`.shell-sidebar__workspace-name`, 16 px uppercase, `--text-secondary`) visually compete with the 80 px-min worktree cards below. The reading order is flat.

**Proposal.** Render workspace names smaller when inactive, and reserve the full-weight label for the active workspace only. Also drop the uppercase on inactive workspaces — uppercase carries too much visual weight for a "not right now" item.

```css
.shell-sidebar__workspace-name {
  font-size: 13px;
  letter-spacing: 0;
  text-transform: none;
  color: var(--text-muted);
}
.shell-sidebar__workspace-name[data-selected="true"] {
  font-size: var(--font-size-label);
  letter-spacing: 0.04em;
  text-transform: uppercase;
  color: var(--text-primary);
}
```

### 2.3 Hover-reveal worktree remove; keep context menu as progressive disclosure — 🟡 medium

**Problem.** Worktree removal is only reachable through a right-click context menu (see `SessionSidebar.tsx:153-167`). Mainstream testers won't find it.

**Proposal.** Add a small `×` affordance that appears on row hover, mirroring the workspace header's `×`. Keep the context menu for power users and future actions. The existing `shell-button--icon shell-button--compact shell-button--round` pattern fits.

### 2.4 Empty worktree list — stronger empty state — 🟢 low

**Problem.** `shell-sidebar__workspace-empty` currently shows "No worktree sessions yet." in plain gray. For a user who just opened a new workspace, the primary call-to-action ("+ New worktree") already exists at the bottom of the group, but the empty state doesn't point to it.

**Proposal.** Replace the empty string with a two-line hint: *"No worktrees yet."* + smaller second line *"Create one below to get started."* Move/keep the `+ New worktree` footer prominent. Also consider promoting it to a primary-styled button when the group is empty.

### 2.5 Process summary pills — make state self-evident — 🟢 low

**Problem.** `.shell-sidebar__process-indicator` communicates `actionRequired` / `active` / `idle` / `exited` by color/fill alone. No tooltip, no text label. The `actionRequired` state is the only one shown in danger red, which is good, but a screen-reader user or a tester in a glare environment loses that.

**Proposal.** Add an `aria-label` / `title` on the indicator reflecting the state name, and consider pairing the dot with a single-char glyph (`●` active, `○` idle, `!` actionRequired, `✕` exited). Keep colors — just don't rely on color alone.

### 2.6 Collapsed rail — redundant first-letter badges — 🟢 low

**Problem.** When collapsed, the workspace badge shows `W` (from "ai-whisper") and worktree items also show `A` (from "ai-whisper" or the worktree label). Two single-letter badges one above the other look like nonsense; it's hard to tell what each means.

**Proposal.** When collapsed, show *only* workspace badges; hide worktree badges and render a thin colored dot-strip underneath each workspace badge to indicate number of worktrees and any attention state. Clicking the badge expands the rail back.

## 3. Main column — session header, terminal section, review stack

### 3.1 Session Info — collapse to one dense line once a session is active — 🔴 high

**Problem.** `SessionHeader.tsx` + `ContextPanel` together render in an expanded "top band" that takes ~120 px vertical: label "SESSION INFO", title `ai-whisper`, a path-group with its own "WORKTREE PATH" label and code block, and a meta strip with Branch / Status / Changes. Every line of this is reference info; the user already sees the active worktree in the rail.

**Proposal.** Default the top band to collapsed (`workspaceState.topBandCollapsed: true`) after first session load. In the collapsed state, `SessionHeader` already renders a one-line strip — it just needs more info: `ai-whisper · master · clean · 0 changes · /Users/vuphan/Dev/ai-whisper`. Let the user expand for the full view (it's already wired with `▴/▾` toggle at `App.tsx:2346`).

This is the single biggest vertical-space win in the app.

### 3.2 Session Note (`ContextPanel.tsx`) — add a real empty state or hide — 🟡 medium

**Problem.** `ContextPanel.tsx` renders a `<textarea>` with a `Session note` label but no `placeholder` attribute, so the field reads as an empty bordered block until the user clicks in and starts typing.

**Proposal.** Two options (pick one):
- **A.** Add a `placeholder="Jot a note for this session…"` to the existing textarea — single-line fix.
- **B.** Collapse the Session Note pane by default when empty, with a tiny "Add note" button in the top band next to the collapse toggle.

I'd pick **A** — lower friction, matches "terminal-first" by not introducing extra chrome. If the top band is collapsed by default (see §3.1), option **A** still works because the note is revealed on expansion.

### 3.3 Review stack — auto-collapse when clean — 🔴 high

**Problem.** `App.tsx:2551-2851`, with `reviewPanelCollapsed` state. When there are no changes (as in the screenshot), the header + Files/Changes/Commits tabs + an empty-viewer message still take ~240 px. That's 20–25% of a typical laptop's vertical viewport spent on a "nothing to see" state.

**Proposal.** When `changes.length === 0` AND `activeSession.reviewMode === "changes"` AND the user hasn't manually expanded the panel, render the header only (collapsed). When changes appear, auto-expand with a gentle animation so the user notices. This preserves the user's manual override if they explicitly expand.

Pseudo-logic:

```tsx
// App.tsx — around the existing reviewPanelCollapsed state
const effectiveCollapsed =
  reviewPanelCollapsed ||
  (userExplicitReviewPanelState === "auto" &&
    activeSession?.reviewMode === "changes" &&
    changes.length === 0);
```

Track whether the last collapse/expand was user-initiated. Persist the rule per worktree.

### 3.4 Review header — surface current-mode count in the title — 🟢 low

**Problem.** "REVIEW: CHANGES" is static. Users have to glance at the tab indicator to know what they're looking at.

**Proposal.** Render `Review: Changes (12)` / `Review: Commits (24 ahead)` / `Review: Files`. Use `--text-muted` for the count so it doesn't compete.

### 3.5 Review tabs — they're buried — 🟡 medium

**Problem.** The Files/Changes/Commits tab bar (`.shell-review-tabs__segments`) sits inside the left rail header (`App.tsx:2639`), so it's scoped to the rail column but affects the viewer column too. It's visually disconnected from the viewer pane it controls.

**Proposal.** Move the segmented control into the review-stack header row (next to `Review:` label and the refresh/collapse icons). This puts the primary control at the top, full-width-aware, and matches what users expect from a three-mode view. The underlying `Tabs.Root` already wraps the grid — just relocate `Tabs.List` upward.

### 3.6 Terminal tabs — close button never appears on tab — 🟡 medium

**Problem.** `TerminalTabs.tsx` renders each tab as just a label. Close is only reachable via context menu. `.shell-terminal-tab__close` is defined in CSS (`shell.css:800`) but never rendered in TSX.

**Proposal.** Show a hover-revealed `×` on each tab when there is more than one tab, and when the tab is the active/focused tab. Wire it to `onClose(process.id)`. Align to the existing `shell-terminal-tab__close` styles.

```tsx
// TerminalTabs.tsx, inside Tabs.Trigger
<span className="shell-terminal-tab__label">{process.label}{suffix}</span>
{processes.length > 1 && (
  <button
    type="button"
    className="shell-terminal-tab__close"
    aria-label={`Close ${process.label}`}
    onClick={(e) => { e.stopPropagation(); onClose(process.id); }}
  >×</button>
)}
```

### 3.7 Terminal utilities — group + label the split toggle — 🟢 low

**Problem.** The split toggle is an unlabeled SVG next to a text "Presets" dropdown and a `+` button. Three adjacent buttons with three different glyph types (SVG, text, text glyph).

**Proposal.** Wrap utility buttons in a visual group (subtle divider between "new shell controls" and "presets"). Add a permanent tooltip on first open, and consider making it a `data-active="true"` visible affordance (blue fill when split mode is on) — currently only the `aria-pressed` changes.

### 3.8 Empty split slots — too wordy — 🟢 low

**Problem.** `App.tsx:2510` / `2522`: *"No shell assigned to this split pane. Use a tab menu to show one here."* is long, and "tab menu" is internal jargon.

**Proposal.** *"Empty pane. Right-click a tab → Show in split left/right."* Shorter, explicit instruction, still one line.

## 5. Dialogs & menus

### 5.1 Fix broken `shell-btn` styles — 🔴 bug, ship-blocker

**Problem.** Three files reference a `shell-btn` class family that doesn't exist in `shell.css`:

- `src/features/viewer/ConfirmCloseDialog.tsx:36,41,48` — Cancel / Discard / Save
- `src/features/viewer/SaveConflictDialog.tsx:37,42,49` — Cancel / Overwrite / Reload
- `src/features/viewer/EditorModal.tsx:319` — Save

The buttons currently render with browser defaults (light-gray system chrome in a dark modal). Confirmed by grep:

```
Grep ".shell-btn" src/app/shell.css  →  No matches found
```

**Proposal — choose one of two paths.**

- **Path A (fast fix)**: Replace `shell-btn` with the existing `shell-button` vocabulary (`shell-button`, `shell-button--danger`; add a `shell-button--primary` if one doesn't yet exist). This keeps the dialog system unified without adding a new class family.
- **Path B (structural fix)**: Define `shell-btn` as a canonical modal-button primitive (smaller, consistent sizing, primary/secondary/danger variants, consistent gap/spacing in `.shell-modal__actions`). Use it everywhere dialogs live.

I recommend **Path A** paired with §5.6 (dialog consolidation).

Also add a missing `gap: var(--space-2)` to `.shell-modal__actions` — currently only `justify-content: flex-end` is set, so without inline margins, buttons would be flush against each other once actual styles apply.

**While you're in `shell.css`:** `.shell-modal__confirm-dirty` (around line 1786) uses two undefined tokens — `var(--text-sm)` (no such token — intended `--font-size-body`?) and `var(--status-danger, #f87171)` (the fallback `#f87171` doesn't match the app's `--danger: #d98c8c`). Both silently fall through to the fallbacks, producing a color mismatch on the dirty checkbox row. Replace with `font-size: var(--font-size-body); color: var(--danger);`.

### 5.2 Editor modal — dirty indicator + Cancel in footer — 🔴 high

**Problem.** `EditorModal.tsx` shows only the file path in the title. There's no visual "you have unsaved changes" affordance other than the disabled state on the Save button (subtle). There's also only one button in the footer (`Save`) and a text-style `Close` in the header; footer conventions expect Cancel + Primary.

**Proposal.**

```tsx
<Dialog.Title className="shell-editor-modal__title">
  {dirty && <span aria-label="Unsaved changes">• </span>}
  {relativePath}
</Dialog.Title>
```

Plus add a `Cancel` / `Discard` button to the footer next to Save, so the primary actions are grouped. Keep the header `×` as secondary dismiss.

Also bump Monaco font-size to match `FileViewer` (`12`, currently `11` in EditorModal at line 24 of `MONACO_OPTIONS`).

### 5.3 `NewWorktreeDialog` — add Cancel, better input affordance — 🟡 medium

**Problem.** `NewWorktreeDialog.tsx` has only "Create worktree" in its action row; dismiss relies on Esc or overlay click. The input has no placeholder or naming-convention hint.

**Proposal.**

```tsx
<div className="shell-modal__actions">
  <button type="button" className="shell-button" onClick={() => onOpenChange(false)}>
    Cancel
  </button>
  <button type="button" className="shell-button shell-button--primary"
          onClick={onConfirm} disabled={!preview || loading || busy}>
    {busy ? "Creating…" : "Create worktree"}
  </button>
</div>
```

Add `placeholder="e.g. feat-auth"` to the name input, and a `--text-muted` helper line *"Used as both the worktree folder and branch name."* under the input. When `loading` (the preview computation), show a compact spinner next to the input, not a bare disabled Create button.

### 5.4 `RemoveWorktreeDialog` — reorder consequence → acknowledgement → action — 🟡 medium

**Problem.** Current order: title → preview (raw `Name:` / `Branch:` / `Path:` lines) → dirty-checkbox → warning copy *"This will remove the linked worktree and force-delete its local branch."* → error banner → Remove button. The warning copy comes after the checkbox, which is backwards: a checkbox should acknowledge a consequence the user has just read, not one they're about to read.

Also: "Running app sessions: none" is inline metadata but in the case where there *are* running sessions, this is a real warning and deserves a banner.

**Proposal.**

```
Title: Remove worktree
[one line summary: Remove ai-whisper (branch feat/auth)]
[path in code style]

[if running sessions: a warning banner listing the processes]
[if isDirty: warning banner "This worktree has uncommitted changes."]

Body copy: "This removes the linked worktree and force-deletes the local branch. This cannot be undone."

[if isDirty: checkbox "I understand I'll lose uncommitted changes"]

[actions:] Cancel | Remove worktree (danger)
```

Add Cancel. Consider making "Remove worktree" require typing the worktree name when dirty *and* running sessions exist (belt + suspenders for the destructive path).

### 5.5 `PresetManager` — full rebuild — 🔴 high

**Problem.** `PresetManager.tsx` is the most inconsistent surface in the app:
- Uses inline `style={{ display: "flex", alignItems: "center", ... }}` on every container.
- Uses native `<button>` elements with no `shell-button` class on Edit / Delete / Launch / Save.
- `Delete` is a single-click destructive action with no confirmation.
- List items don't visually separate from the editing form below.
- No Enter-to-save on the form; label/command inputs don't form a `<form>`.
- No cancel button on the in-progress edit; a user mid-edit has no way out.
- Empty state is implicit (no presets → list is simply absent).

**Proposal.** This deserves a real rewrite rather than patches. Target shape:

```
Title: Command presets
Copy: "Launch a named shell with a saved command."

[list, one row per preset:]
  [launch button] [label] [code command, ellipsized] [edit icon] [delete icon]

[divider]

[form:]
  Preset label  [input, placeholder: e.g. dev server]
  Command       [input, placeholder: pnpm dev, autofocus if no presets]
  [actions:] Cancel edit | Save preset

[empty state, when no presets:]
  "No presets yet. Save a frequently-used command below to launch it in one click."
```

Implementation notes:
- Wrap label/command inputs in a `<form>` with `onSubmit={handleSave}` so Enter works.
- Confirm on delete: either inline "Delete? · Confirm / Cancel" or a nested `ConfirmCloseDialog`-style modal.
- Replace all inline `style={{}}` with classes (`shell-preset-dialog__list`, `shell-preset-dialog__row`, etc.).
- Use the standard `shell-button` vocabulary with `shell-button--primary` for Save.
- Add a leading "Launch" icon (▶) to the launch button so users don't have to read a whole button.
- Make the edit/delete icons right-aligned and hover-reveal only, to reduce clutter on a list of many presets.

### 5.6 Dialog consolidation — one primitive, three sizes — 🟡 medium

**Problem.** Four distinct modal systems exist:

| System | Used by | Overlay z-index | Content class |
|---|---|---|---|
| `shell-modal*` | New/Remove/Load worktree, DiscardChange (presumed) | 51 | `shell-modal`, `shell-modal--worktree`, `shell-modal--workspace-picker` |
| `shell-md-modal*` | MarkdownPreviewModal | n/a (uses `shell-md-overlay`) | `shell-md-modal` |
| `shell-editor-modal*` | EditorModal | n/a (uses `shell-editor-overlay`) | `shell-editor-modal` |
| `shell-preset-dialog*` | PresetManager | n/a (uses `shell-preset-overlay`) | `shell-preset-dialog` |

They're doing the same job (Radix `Dialog.Root` + overlay + content) with different class namespaces and slightly different z-indices/styling. Maintenance cost will compound.

**Proposal.** Extract `shared/ui/Modal.tsx` with `size` prop (`"sm" | "md" | "lg" | "xl"`) backed by one CSS family (`shell-modal`, `shell-modal--sm`, etc.). Have every dialog consume it. Keep Radix as the primitive. Migration can happen piecemeal — start with the three `shell-btn`-fix dialogs since you're touching them anyway.

### 5.7 `LoadWorkspaceDialog` — show recents — 🟡 medium

**Problem.** Opening the dialog drops users directly into a path input with no recency. Users who just used a workspace an hour ago have to remember or re-select the path.

**Proposal.** Under the input, render a list of recently opened workspaces (already implicit in `app-workspaces-state.ts`). One-click to open, with the full path on hover. Cap at 10 entries.

### 5.8 Confirm dialogs — button-order convention — 🟢 low

**Problem.** `ConfirmCloseDialog` order: Cancel | Discard | Save. `SaveConflictDialog` order: Cancel | Overwrite | Reload. The destructive action (Discard, Overwrite) sits immediately next to the primary — an easy accidental click.

**Proposal.** macOS-style order: `Discard | spacer | Cancel | Save` — destructive on the left, safe primary on the right, with a visual gap between destructive and non-destructive. Use `flex: 1` on a spacer or `margin-right: auto` on Discard. Apply to both dialogs.

### 5.9 Context menus — discoverability — 🟢 low

**Problem.** Several high-value actions live only in right-click context menus: worktree removal, terminal session Stop/Restart/Pin, split-show-in-left/right, file Preview, file Edit. Mainstream devs often don't right-click.

**Proposal.** Pair every context-menu action with at least one of: hover-revealed icon, toolbar button, keyboard shortcut, or a kebab (`⋮`) menu on the row. Sprinkling affordances widely is cheaper than retraining users.

## 6. Viewer & editor surfaces

### 6.1 `WorktreeTree` — add a header toolbar — 🔴 high

**Problem.** The tree has a search input but no visible refresh button (only a context-menu "Refresh" on the invisible root row — essentially hidden). No "Collapse all" or "Expand all". The search input has no clear button. The tree has inline `style` overrides (`marginLeft: 8px` on the wrapper, inline styles on row labels).

**Proposal.** Add a small toolbar above the tree:

```
[  search input                    ✕  ]  [↻ refresh] [⇱ collapse all]
```

- Wire refresh to the existing `reload()` function.
- Clear button on search shows when `inputTerm.length > 0`.
- "Collapse all" sets `expandedPaths` to just `[WORKTREE_TREE_ROOT_PATH]`.
- Replace all inline `style={{}}` with CSS classes.

### 6.2 Tree rows — differentiate files from folders visually — 🟢 low

**Problem.** Folders have a chevron, files have nothing in that slot — so the filename start-columns don't align between files and their sibling folders.

**Proposal.** Reserve the chevron space for files with an invisible spacer, so names align. Optionally dim file icons (a minimal file-type glyph) to help skim — though this conflicts with the terminal-first minimalism, so consider it optional.

### 6.3 Git status badges — tooltip the raw characters — 🟢 low

**Problem.** `M` / `A` / `D` / `??` / `R` git status chars are dev-literate but not self-explanatory.

**Proposal.** Add `title="Modified"`, `title="Added"`, `title="Deleted"`, `title="Untracked"`, `title="Renamed"` on the badge. Zero layout cost.

### 6.4 `FileViewer` + `DiffViewer` — action bar, not just title — 🟡 medium

**Problem.** Both viewers have a header showing only the path, with file actions hidden in right-click. No "Copy path", "Open in editor", "Reveal in Finder", "Copy file contents" affordances.

**Proposal.** Add a compact action bar on the right side of `shell-viewer__header`:

```
[path ellipsized]                         [Edit] [Copy path] [⋮]
```

- "Edit" appears only when `isEditable(basename)` — replaces the context-menu-only entry point.
- "Copy path" is a tiny icon button; one-click UX for an extremely common action.
- Kebab menu holds lower-frequency actions (Preview for `.md`, Reveal in Finder, etc.).

Also: `FileViewer` currently shows `Error: {message}` — drop the `Error:` prefix since the style already communicates error via `.shell-error`.

### 6.5 Monaco font-size consistency — 🟡 medium

**Problem.** Three Monaco instances with two different font sizes:
- `FileViewer.tsx:140` — `fontSize: 12`
- `DiffViewer.tsx:57` — `fontSize: 12`
- `EditorModal.tsx:24` — `fontSize: 11`

**Proposal.** Pick one (12 reads slightly better at 13" laptop sizes, matches the app body at 13 px). Extract `MONACO_BASE_OPTIONS` into `shared/editor/` (there's already a folder there) and import in all three places.

### 6.6 Language detection — extract shared map — 🟢 low

**Problem.** `DiffViewer.tsx` has `EXTENSION_TO_LANGUAGE` and `EditorModal.tsx` has `languageForBasename`. Two near-duplicate implementations.

**Proposal.** Move to `shared/editor/language-from-path.ts` and import from both. Bonus: also supports `.toml`/`.xml`/`.scss` that currently diverge between the two.

### 6.7 `MarkdownPreviewModal` — retry UX — 🟢 low

**Problem.** On load error, the Retry button sits inline with no spacing, uses `shell-md-modal__retry` which is a unique class elsewhere unused. If the user hits Retry and it fails again, there's no feedback that anything happened.

**Proposal.** Replace with a centered empty-state card showing an icon, message, and a `shell-button shell-button--compact` "Retry" button. Show a brief "Retrying…" state while the reload runs.

### 6.8 `EditorModal` — save-conflict flow has a subtle bug surface — 🟢 low

**Problem.** When the user hits Overwrite on a save conflict, the dialog closes instantly; if the overwrite later fails (permission denied, disk full), the error is surfaced via `status` — a small footer message that auto-clears in 3 s.

**Proposal.** After an overwrite/save failure triggered from a dialog, keep the status message sticky until dismissed or until the next save attempt. Auto-clear is fine for success; not for error.

## 7. Cross-cutting items (still within scope)

### 7.1 Standardize expand/collapse + close glyphs — 🟡 medium

**Problem.** The app uses all of these today:

| Glyph | Where |
|---|---|
| `▸ / ◂` | Sidebar collapse/expand toggle (`SessionSidebar.tsx:54`) |
| `▴ / ▾` | Top band collapse (`App.tsx:2363`), review panel collapse (`App.tsx:2609`), tree directory chevron (`WorktreeTree.tsx:146`) |
| `×` | Workspace remove, restore-warning dismiss, editor-modal close-button target states |
| `✕` | `MarkdownPreviewModal.tsx:61` |
| text "Close" | `EditorModal.tsx:298` |

**Proposal.** One triangle set (`▴▾` for vertical collapse, `▸▾` for tree/horizontal expansion) and one `×` (pick the `×` U+00D7 already in use — drop `✕`). Kill the text "Close" button in EditorModal — the header already has an `×`.

### 7.2 Button size + hit target consistency — 🟢 low

**Problem.** `shell-button--compact` is 32 px tall and `shell-button--icon` is 32 × 32. But many smaller `×` / chevron buttons use no class and end up 24 px or even smaller, inconsistent across the app.

**Proposal.** Audit all `<button>` elements and give every interactive control at least 28 × 28 effective hit area (transparent padding is fine). Define a `shell-button--xs` variant for dense rows (tree search clear, workspace `×`) at 24 × 24 to avoid the 32 px buttons feeling oversized in the rail.

### 7.3 Keyboard shortcut baseline — 🔴 high

**Problem.** Global key handling exists only for `Cmd+E` (open editor — `App.tsx:478`). Editor-local handling adds `Cmd+S` (save — `EditorModal.tsx:252`). That's it. A terminal-first app shipping without basic shortcuts feels like a demo.

**Proposal — minimum shortcut set for beta.**

| Shortcut | Action |
|---|---|
| `Cmd/Ctrl + B` | Toggle sidebar |
| `Cmd/Ctrl + T` | New shell (ad-hoc) |
| `Cmd/Ctrl + W` | Close active shell (if > 1) |
| `Cmd/Ctrl + 1..9` | Switch to nth shell |
| `Cmd/Ctrl + \` | Toggle split mode |
| `Cmd/Ctrl + O` | Open workspace |
| `Cmd/Ctrl + Shift + R` | Refresh review / tree |
| `Cmd/Ctrl + ?` or `Cmd/Ctrl + /` | Show shortcut cheat sheet |

Ship a `ShortcutsOverlay` modal bound to the last one. Put the open shortcut in the "empty worktree" and "empty shell" states as discovery help.

### 7.4 Restore warning banner — clearer affordance — 🟢 low

**Problem.** `.shell-restore-warning` at `App.tsx:2282` is a `role="status"` banner with a tiny `×` dismiss. No type (info/warn/error) signal via icon. On a dark theme with muted colors, it reads as decorative.

**Proposal.** Add an icon/glyph (`⚠`) and an action link if relevant (e.g. "Re-open last workspace"). Differentiate from errors — warnings should use `--warning`, not `--danger` backgrounds.

### 7.5 Focus rings — present but subtle — 🟢 low

**Problem.** Focus-visible rules at `shell.css:421-426` apply `outline: 2px solid var(--accent); outline-offset: 2px` to `.shell-note-input`, `.shell-button`, `[role="tab"]`. That's good, but many custom components (`.shell-sidebar__item`, `.shell-list__item`, `.shell-terminal-tab__close`, tree rows) don't inherit this.

**Proposal.** Add a single global rule: `:focus-visible { outline: 2px solid var(--accent); outline-offset: 2px; border-radius: inherit; }` scoped to interactive elements, and remove the per-component duplicates. Makes the whole app keyboard-navigable with one rule.

### 7.6 Empty-state voice — 🟢 low

**Problem.** Empty-state copy across the app is inconsistent in voice:
- "Select a file or changed file to inspect it." (friendly)
- "No changed files." (blunt)
- "No shell assigned to this split pane. Use a tab menu to show one here." (passive, jargon)
- "Loading files…" (ellipsis style)
- "No presets yet" (separator omitted — shell.css has `shell-toolbar-menu__item--disabled` rule using it, good)

**Proposal.** A UX-copy sweep — normalize to: short declarative first sentence + optional actionable second sentence. I'd recommend a focused pass with the `design:ux-copy` skill once structural changes land.

These came up during review but land outside the agreed scope. Flag if any should be pulled in:

- **Design tokens & motion.** `--space-*` / `--radius-*` are well-factored but there are several inline magic numbers (8 px, 180 px, 16 px in `WorktreeTree`, 240 px in `shell-main-column`). A token audit + a motion/transition pass (currently four `transition:` rules in `shell.css` using only two unique durations, 120 ms and 150 ms) would tighten things further.
