# ai-14all — v0.1.0 Release Design

Spec for graduating ai-14all from `v0.1.0-beta.N` to the first stable `v0.1.0` release. Covers distribution, update-notification, release pipeline, repo flip, and documentation. Assumes the `2026-04-23-redesign.md` work is complete on master.

## 1. Purpose

Ship `v0.1.0` as the first non-beta release of ai-14all to a small circle of developer friends. The release is publicly downloadable from `ai-creed.dev`, but rollout is friend-scope — no marketing push, no mass onboarding.

Three product commitments frame this release:

- **No new features.** v0.1.0 is the current master snapshot after the redesign, not a feature push. All scope goes into distribution infrastructure.
- **Unsigned is acceptable for now.** Apple Developer enrollment is deferred. v0.1.0 documents the right-click-Open workaround; signing lands in a later patch.
- **Notify-only updates.** The app tells users when a newer version exists, then routes them to the browser. In-app install is a post-signing upgrade.

## 2. Scope

### In scope

- Cut version `0.1.0` from current master.
- Host release artifacts on Cloudflare R2 behind `downloads.ai-creed.dev`.
- Publish `latest-mac.yml` manifest via ai-creed's Astro/Pages site at `https://ai-creed.dev/ai-14all/latest-mac.yml`.
- Ship in-app notify-only updater: main-process manifest check on launch, dismissible banner in the renderer, browser hand-off for download.
- Automate the release via GitHub Actions on stable tag push.
- Flip `ai-creed/ai-14all` repo from private to public, add MIT license.
- Refresh user-facing documentation: `README.md`, new `CHANGELOG.md`, new `KNOWN-ISSUES.md`, archived `beta-testing.md`.
- Expand the ai-14all project page on ai-creed.dev with a Download section and status flip to `stable`.
- Add a new `scripts/release-stable.mjs` alongside the existing beta script, sharing neither tests nor flow beyond style.
- Documentation of versioning policy for post-v0.1.0 releases.

### Out of scope

- Code signing / notarization (tracked as a known issue; target v0.1.1 once Apple Developer enrollment is done).
- In-app auto-install of updates (target v0.2.0 after signing).
- Intel Mac, Windows, Linux builds. Apple Silicon arm64 only.
- Crash reporting, telemetry, analytics. Local log files only (existing Electron default path).
- A "latest" URL alias on R2 that always points to the newest DMG. Download links are versioned.
- CI coverage of every PR. This spec adds a release-only workflow; PR gating stays at local `pnpm test:all` unless separately decided.
- CONTRIBUTING.md, CLA, code of conduct. Friend scope does not justify them.
- ai-creed website restructure beyond the ai-14all project page. The site is live and functional.

## 3. Product contracts

### 3.1 Artifact layout

Per release (produced by `pnpm package:mac`):

| Artifact | Destination | Canonical URL |
|---|---|---|
| `ai-14all-<version>-arm64.dmg` | R2 bucket `ai-14all-releases` | `https://downloads.ai-creed.dev/ai-14all/<version>/ai-14all-<version>-arm64.dmg` |
| `ai-14all-<version>-arm64-mac.zip` | same R2 bucket | `https://downloads.ai-creed.dev/ai-14all/<version>/ai-14all-<version>-arm64-mac.zip` |
| `latest-mac.yml` | ai-creed repo at `public/ai-14all/latest-mac.yml` | `https://ai-creed.dev/ai-14all/latest-mac.yml` |

R2 layout preserves every shipped version under `ai-14all/<version>/`. No lifecycle expiration.

### 3.2 Update manifest schema

`latest-mac.yml` mirrors electron-updater's native shape so a future auto-install upgrade is a drop-in change.

electron-builder already emits `latest-mac.yml` during `pnpm package:mac` without a `publish` block (verified against `release/latest-mac.yml` from the last beta build). The generated file uses **relative** `url` values and selects the **ZIP** as the top-level `path`:

```yaml
# as emitted by electron-builder — do not publish this verbatim
version: 0.1.0
releaseDate: '2026-04-24T10:00:00.000Z'
path: ai-14all-0.1.0-arm64-mac.zip
sha512: <base64-sha512-of-zip>
files:
  - url: ai-14all-0.1.0-arm64-mac.zip
    sha512: <base64-sha512-of-zip>
    size: <bytes>
  - url: ai-14all-0.1.0-arm64.dmg
    sha512: <base64-sha512-of-dmg>
    size: <bytes>
```

The notify updater serves a single flat manifest URL (`https://ai-creed.dev/ai-14all/latest-mac.yml`) with no per-version base URL context, so relative URLs cannot be resolved at runtime. A CI rewrite step (§4.3) transforms the emitted manifest into the **published** form before copying it into ai-creed:

```yaml
# published form after CI rewrite — what the notifier consumes
version: 0.1.0
releaseDate: '2026-04-24T10:00:00.000Z'
path: https://downloads.ai-creed.dev/ai-14all/0.1.0/ai-14all-0.1.0-arm64.dmg
sha512: <base64-sha512-of-dmg>
files:
  - url: https://downloads.ai-creed.dev/ai-14all/0.1.0/ai-14all-0.1.0-arm64.dmg
    sha512: <base64-sha512-of-dmg>
    size: <bytes>
  - url: https://downloads.ai-creed.dev/ai-14all/0.1.0/ai-14all-0.1.0-arm64-mac.zip
    sha512: <base64-sha512-of-zip>
    size: <bytes>
```

The rewrite:

- Prepends `https://downloads.ai-creed.dev/ai-14all/<version>/` to every `files[].url`.
- Replaces the top-level `path` with the DMG URL (not the ZIP), because the banner's Download button must hand a DMG to the browser. The top-level `sha512` is re-sourced from the DMG entry in `files[]`.
- Leaves `version`, `releaseDate`, `size`, and per-file `sha512` untouched.

Runtime validation (§3.3) enforces absolute URLs under the canonical download host on the consumer side, guarding against accidental publication of an un-rewritten manifest.

### 3.3 Notify updater behavior

Main process (`src/main/services/updateNotifier.ts`):

- Fires once per launch, 3 seconds after `app.whenReady()`, so startup is unaffected.
- Fetches `https://ai-creed.dev/ai-14all/latest-mac.yml` with a 5 second timeout.
- Parses yaml (new runtime dep: `js-yaml`).
- Validates:
  - `version` matches `^\d+\.\d+\.\d+$` exactly — strict three-segment semver, no prerelease suffix, no build metadata. This is deliberately stricter than full SemVer: the notifier only advertises stable releases, so prerelease or suffixed manifests are treated as invalid and silently dropped. The smoke manifest in §9.3 uses a `-smoke` suffix specifically so that any live app that briefly fetches it during the smoke window fails this check and never surfaces a banner.
  - `path` and every `files[].url` start with `https://downloads.ai-creed.dev/`.
  - Required fields present: `version`, `path`, `files`, `releaseDate`.
- Compares manifest version with `app.getVersion()` via a local semver compare helper. No new dep.
- If the manifest version is strictly greater, emits an IPC event via `webContents.send('update:available', { version, url, releaseDate })`.
- Any failure path — network, parse error, schema rejection, allow-list miss — logs at `warn` and exits silently. The banner must never appear on invalid data.
- Disabled in dev: the notifier only runs when `app.isPackaged === true`. Tests can force it on via a DI seam.

Preload contract addition (`electron/preload/index.ts` + `shared/contracts/commands.ts`):

The existing preload exposes a typed `Ai14AllDesktopApi` on `window.ai14all`. Add a new `system` namespace (or extend it if it exists) with two members:

- `window.ai14all.system.onUpdateAvailable(cb: (info: UpdateInfo) => void): UnsubscribeFn` — subscribes to the `update:available` IPC event emitted by the notifier.
- `window.ai14all.system.openExternal(url: string): Promise<void>` — main-process handler validates the URL host equals `downloads.ai-creed.dev` before calling `shell.openExternal`. Any other host is rejected. No other preload exposes `shell.openExternal`, so this addition is the single canonical surface.

`UpdateInfo` joins `shared/contracts/commands.ts` alongside the existing command types.

**E2E and unit test seam (main process).**

The renderer has no visibility into the main-process fetch, so Playwright's `page.route` cannot stub it. The notifier must accept two test-only overrides, both gated on `process.env.AI14ALL_E2E === "1"` so production builds are unaffected:

- `AI14ALL_E2E_UPDATE_MANIFEST_URL` — when set, the notifier fetches the manifest from this URL instead of the canonical ai-creed URL. Tests point it at a `file://` fixture path or a short-lived local HTTP server.
- `AI14ALL_E2E_UPDATE_FORCE` — when set to `1`, the notifier skips all network fetching and emits a fixed `update:available` payload constructed from `AI14ALL_E2E_UPDATE_VERSION`, `AI14ALL_E2E_UPDATE_URL`, and `AI14ALL_E2E_UPDATE_RELEASE_DATE`. Used for tests that only need the banner UI flow.

Unit tests import the notifier's pure functions (`shouldNotify`, `validateManifest`) directly, no env required.

Renderer banner (`src/features/updater/UpdateBanner.tsx`):

- Rendered above the session chip bar, full-width, approximately 36 px tall.
- Content: `v{newVersion} available` + `Download` anchor + close icon.
- Clicking `Download` calls `window.ai14all.system.openExternal(url)` with the URL from the IPC payload.
- Close dismisses the banner for the current session only; state is not persisted. Next launch re-checks and re-shows if still eligible.
- Dark and light theme parity required.
- Respects reduced motion: banner fades/slides optional, instant entrance for reduced-motion users.
- Accessible name: `role="status"`, `aria-live="polite"`.

### 4.3 GitHub Actions workflow

File: `.github/workflows/release.yml`.

Trigger:

```yaml
on:
  push:
    tags:
      - 'v*.*.*'
      - '!v*-*'
  workflow_dispatch:
    inputs:
      ref:
        description: 'Tag or branch to release from'
        required: true
      version:
        description: 'Override version (required for smoke runs with no tag)'
        required: false
```

GitHub Actions tag filters use **glob patterns with include + exclude pairs**, not regex. The include pattern `v*.*.*` matches every three-segment tag; the exclude pattern `!v*-*` drops any tag with a hyphen, which excludes all `-beta.N`, `-rc.N`, or other prerelease suffixes. The two lines together match stable releases only.

`workflow_dispatch` accepts an optional `version` input. When invoked via tag push, `version` is empty and the workflow derives the version from `GITHUB_REF_NAME` with the leading `v` stripped. When invoked via `workflow_dispatch` for a smoke run on a branch (no tag), `inputs.version` is required and becomes the source of truth.

Runner: `macos-14` (Apple Silicon, free minutes on public repos).

Sequential steps:

1. Resolve `VERSION`: if `inputs.version` set, use that; else `${GITHUB_REF_NAME#v}`. Resolve `RESOLVED_REF`: `inputs.ref || github.ref`.
2. Checkout at `RESOLVED_REF`.
3. Setup Node 22 + pnpm via corepack.
4. `pnpm install --frozen-lockfile`.
5. `pnpm lint`, `pnpm format`, `pnpm typecheck`.
6. `pnpm test`.
7. `pnpm test:e2e`.
8. **Align `package.json.version` with `VERSION`.** electron-builder reads `package.json.version` to name artifacts and populate the emitted manifest. Tag-push runs are expected to already match (the stable script commits the bump before tagging), but always writing removes the failure mode where a smoke run packages the wrong version. The workflow rewrites `package.json` in-place, does **not** commit the change, and relies on the ephemeral checkout being discarded after the job. Mismatch guard: if `RESOLVED_REF` is a tag and the pre-rewrite `package.json.version` already differs from `VERSION`, fail immediately — that indicates a stale tag pointing at the wrong commit.
9. `pnpm package:mac` — emits `release/ai-14all-<VERSION>-arm64.dmg`, `release/ai-14all-<VERSION>-arm64-mac.zip`, `release/latest-mac.yml`.
10. **Pre-rewrite validation.** Parse `release/latest-mac.yml`. Assert `version === VERSION` and that per-file `sha512` entries are non-empty. Fail loudly if the emitted version does not match — that indicates the step-8 alignment silently failed.
11. **Manifest rewrite.** Transform the emitted manifest per §3.2 into the published form:
    - Prefix every `files[].url` with `https://downloads.ai-creed.dev/ai-14all/${VERSION}/`.
    - Replace the top-level `path` with the DMG URL and the top-level `sha512` with the DMG's `sha512` (sourced from the matching `files[]` entry).
    - Preserve `version`, `releaseDate`, `files[].size`, and per-file `sha512`.
    - Write the result to `release/latest-mac.published.yml` (keep the raw emitted file for debugging).
12. **Post-rewrite validation.** Re-parse `release/latest-mac.published.yml`. Assert every `files[].url` and the top-level `path` start with `https://downloads.ai-creed.dev/ai-14all/${VERSION}/`.
13. Upload DMG + ZIP to R2 via `aws s3 cp` with `--endpoint-url https://${R2_ACCOUNT_ID}.r2.cloudflarestorage.com`, target prefix `s3://ai-14all-releases/ai-14all/${VERSION}/`. Idempotent; reruns overwrite.
14. Checkout the ai-creed repo with the deploy SSH key, copy `release/latest-mac.published.yml` to `public/ai-14all/latest-mac.yml`, `git pull --rebase origin master`, commit `chore(releases): ai-14all v${VERSION}`, push.
15. Create or refresh a GitHub Release on ai-14all for the tag (skipped if `inputs.version` was used and no tag exists — smoke runs do not create GitHub Releases). The exact command sequence, so reruns are idempotent:
    ```bash
    if gh release view "vX.Y.Z" >/dev/null 2>&1; then
      gh release upload "vX.Y.Z" release/*.dmg release/*.zip --clobber
    else
      gh release create "vX.Y.Z" release/*.dmg release/*.zip \
        --title "vX.Y.Z" \
        --notes "See CHANGELOG.md for release notes."
    fi
    ```
    First run creates the release and uploads assets. Reruns hit the `view` branch and refresh assets via `upload --clobber` without erroring on the existing release.

Failure handling:

- **Any step 1–10 fails.** No artifacts uploaded, nothing on ai-creed. If a release tag was already pushed, the tag and its release commit on master are now stale. Roll back both:
  ```
  git push origin :refs/tags/vX.Y.Z    # delete remote tag
  git tag -d vX.Y.Z                    # delete local tag
  git revert --no-edit <release-commit-sha>   # add a revert commit on master
  git push origin master
  ```
  Then fix the underlying issue and rerun the stable script.
  An alternative (only if you are the sole contributor to master and no one else has pulled the bump commit): `git reset --hard HEAD~1 && git push --force origin master`. Use the revert path in any shared-history scenario.
- **Manifest rewrite (step 11) or post-rewrite validation (step 12) fails.** Bug in the rewrite script. Fix the script, rerun the workflow via `workflow_dispatch` on the same ref. Safe: no network work happened yet.
- **R2 upload (step 13) partial.** Rerun the workflow. `aws s3 cp` overwrites existing keys; no cleanup needed.
- **ai-creed commit (step 14) fails.** Rerun the workflow — step 14 begins with `git pull --rebase` so it self-heals against concurrent pushes. If the conflict is structural, resolve in a separate PR on ai-creed.
- **GitHub Release creation (step 15) fails.** Rerun via `workflow_dispatch` with the same tag. The step 15 `gh release view || gh release create` pattern makes reruns idempotent — no manual cleanup of a partially-created release is required.

## 7. Versioning policy

| Bump | When | Example |
|---|---|---|
| patch | Bug fixes, unsigned-to-signed transition with no feature change | 0.1.0 → 0.1.1 |
| minor | New feature, backwards-compatible UI change | 0.1.0 → 0.2.0 |
| major | Breaking workflow change, v1.0 graduation | 0.x → 1.0 |

Tag patterns are disjoint so CI only fires on stable:

| Track | Pattern | Workflow |
|---|---|---|
| beta | `v0.1.0-beta.14`, `v0.2.0-beta.1` | None. Local `release-beta.mjs` only. |
| stable | `v0.1.0`, `v0.1.1`, `v0.2.0` | `.github/workflows/release.yml` |

The beta script and the stable script are siblings. They do not share code paths beyond style. The beta script keeps the "run full local gauntlet before tagging" discipline because that is where failures are cheapest to catch. The stable script delegates validation to CI and keeps local work fast.

## 8. Release pipeline scripts

### 8.1 `scripts/release-stable.mjs`

Responsibilities (local):

1. Enforce a clean working tree.
2. Enforce the current branch is `master`.
3. Accept CLI input and compute `TARGET_VERSION`:
   - `node scripts/release-stable.mjs patch` → bump the patch segment of the stable part of `package.json.version`. For the first stable cut (current version `0.1.0-beta.14`), `patch` is **rejected** with a clear error pointing at `--version`.
   - `node scripts/release-stable.mjs minor` → bump minor, reset patch. Same rejection on the first stable cut.
   - `node scripts/release-stable.mjs major` → bump major, reset minor and patch. Same rejection on the first stable cut.
   - `node scripts/release-stable.mjs --version X.Y.Z` → explicit version. Required path for the first stable cut from `0.1.0-beta.14`.
4. Enforce that `TARGET_VERSION` is strict semver `^\d+\.\d+\.\d+$` — no prerelease, no build metadata, no `-beta`. This guard applies to the **target**, not the current `package.json.version`, which may still carry `-beta.N` during the first stable cut.
5. **Local pre-flight** before any git state change: run `pnpm install --frozen-lockfile`, `pnpm typecheck`, `pnpm test`, and `pnpm package:mac`. If any step fails, exit immediately with no commits, no tags, no pushes. This catches obvious breakage before master gets a release commit.
6. Write `TARGET_VERSION` into `package.json`, commit `chore: release vX.Y.Z`.
7. Create an annotated tag: `git tag -a vX.Y.Z -m "Release vX.Y.Z"`. Annotated (not lightweight) so `--follow-tags` pushes it.
8. Push the branch and the tag in one invocation (`git push --follow-tags`).
9. Print the expected CI URL for easy follow-up and remind the operator that CI failure requires tag + commit rollback per §4.3.

Testing: unit coverage mirrors the existing beta script tests — target-version computation for each CLI mode including the first-cut bootstrap case, `package.json` rewriter, clean-tree / branch / target-semver guards, and CLI arg parser. The pre-flight step is exercised only via manual testing; it is a thin `execFileSync` wrapper around existing scripts and needs no dedicated harness.

### 8.2 `scripts/release-beta.mjs`

Unchanged. Continues to handle the beta track locally with its existing full local test gauntlet.

## 9. Rollout order

Order matters so partial state never blocks a real cut.

1. **Code-only prep, behind no flag.**
   - `UpdateBanner`, `updateNotifier`, preload contract additions, unit + e2e tests. Notifier is gated on `app.isPackaged`, so dev is unaffected.
   - `scripts/release-stable.mjs` + unit tests.
   - `.github/workflows/release.yml`, including `workflow_dispatch` so the first invocation can be manual.
   - `LICENSE`, drafts of `README.md`, `CHANGELOG.md`, `KNOWN-ISSUES.md`.
   - `electron-builder.yml` needs no changes. electron-builder already emits `latest-mac.yml` during `pnpm package:mac` without a `publish` block; the CI rewrite step in §4.3 produces the published form that the notifier consumes.

2. **Infra prep, outside the repo.**
   - Create R2 bucket + custom domain.
   - Generate R2 API token, add three GitHub secrets.
   - Generate deploy SSH key, add private half as `AI_CREED_DEPLOY_KEY`, public half to ai-creed.
   - Commit `public/ai-14all/.gitkeep` to ai-creed. Confirm Cloudflare Pages picks up the directory.

3. **CI dry run.**
   - Create a short-lived branch `smoke/release-<date>` off master with no version bump and no tag.
   - Trigger the workflow via `workflow_dispatch` with `inputs.ref = smoke/release-<date>` and `inputs.version = 0.0.1-smoke`. The workflow uses that input as the source of truth for the version; no tag is required because the trigger is `workflow_dispatch`, not tag push.
   - The stable-tag trigger and the GitHub Release step are skipped on this run: step 15 checks whether the run originated from a tag push before calling `gh release create`.
   - Verify: DMG + ZIP land at `downloads.ai-creed.dev/ai-14all/0.0.1-smoke/`; `public/ai-14all/latest-mac.yml` commit appears on ai-creed master with `version: 0.0.1-smoke` and absolute URLs under `0.0.1-smoke/`; no GitHub Release is created on ai-14all.
   - Clean up:
     - Revert the ai-creed smoke commit (`git revert <commit>`) so the published manifest does not advertise the smoke version to any launched app.
     - Delete the R2 objects under `ai-14all/0.0.1-smoke/`.
     - Delete the branch.
   - The smoke version `0.0.1-smoke` is intentionally rejected by the notifier's strict `^\d+\.\d+\.\d+$` validator (§3.3). That is the point: during the brief window between the workflow's ai-creed push and the revert, any live app that fetches the smoke manifest drops it on validation and shows no banner. The revert is still required (it is the primary safeguard), but the strict validator is defense in depth against a forgotten cleanup.

4. **Flip the repo public + expand the ai-creed project page + merge the doc rewrites.**

5. **Cut v0.1.0.**
   - `node scripts/release-stable.mjs --version 0.1.0` → tag pushed → CI runs.
   - Confirm the full pipeline succeeds end-to-end.
   - Open a packaged DMG on a friend's Mac (or a clean admin account) to confirm install works.

6. **Ping friends.** Share the ai-creed download URL.

## 10. Verification expectations

Every shipped slice in this spec must include:

- **Unit tests.**
  - `release-stable.mjs`: target-version computation for each CLI mode including the first-cut bootstrap from `0.1.0-beta.14`, `package.json` rewriter, clean-tree / branch / target-semver guards, CLI arg parser.
  - `updateNotifier`: semver compare edge cases (equal, newer, older, prerelease against stable, invalid inputs); manifest validator accepts valid rewritten shape, rejects missing fields, rejects wrong host, rejects bad semver; `openExternal` handler rejects URLs outside the `downloads.ai-creed.dev` host.
  - Manifest rewrite helper used by the CI step: given a freshly emitted electron-builder yml, produces the absolute-URL form with DMG as `path`. Round-trips the `version`, `releaseDate`, and per-file `sha512` fields unchanged.
- **E2E tests (Playwright + Electron).**
  Existing tests launch the packaged Electron build with `AI14ALL_E2E=1` and related env vars (see `tests/e2e/cumulative-flow.phase-10.test.ts`). The main-process fetch is invisible to the renderer, so update-flow tests use the main-process seams defined in §3.3, not `page.route`.
  - **Banner appears on newer version.** Write a fixture manifest to a temp `file://` URL (or serve it via an ephemeral `http.createServer` on `127.0.0.1`). Launch with `AI14ALL_E2E_UPDATE_MANIFEST_URL` pointing at that URL. Assert the banner is visible and contains the manifest version.
  - **Banner hidden when current.** Same seam, manifest version matches `app.getVersion()`. Assert the banner never renders within a reasonable timeout.
  - **Banner hidden on invalid manifest.** Same seam, fixture has bad host or missing fields. Assert the banner never renders and main-process logs contain the validation warning.
  - **Download hand-off.** Launch with `AI14ALL_E2E_UPDATE_FORCE=1` and the companion env overrides so the banner appears deterministically without any network. Intercept `window.ai14all.system.openExternal` via a test-only preload shim guarded by `AI14ALL_E2E=1` that pushes invocations onto a global test array. Assert the array contains the expected DMG URL after clicking `Download`.
  - **Dismiss within session.** Click the close control, assert banner hidden. Reload the window (not the whole Electron app), assert banner re-renders if the fixture still advertises a newer version — this confirms dismissal is session-scoped renderer state, not persisted.
- **Manual smoke before first real cut.**
  - CI dry run via `workflow_dispatch` on a short-lived smoke branch with `inputs.version = 0.0.1-smoke` (covered by §9.3).
  - Install the v0.1.0 DMG on a clean admin account, right-click-Open, verify launch.
  - Confirm banner absent on that build (no newer manifest yet).
  - After cutting a fake v0.1.1 manifest manually, relaunch and confirm banner appears and download handoff works.
- **Dark and light theme check on the banner.**
- **Keyboard path for banner dismissal and download.** Tab to the banner, `Enter` on `Download`, `Enter` on the close control.

## 11. Open follow-ups

Tracked here so they are not forgotten but are explicitly outside v0.1.0:

- **Signing and notarization.** Apple Developer enrollment, Developer ID Application certificate, entitlements plist, notarytool integration, CI secret for signing identity. Targets v0.1.1 as a feature-free patch.
- **In-app auto-install.** Flip `updateNotifier` into a full `electron-updater` integration once builds are signed. Targets v0.2.0 as a minor release. Schema already compatible.
- **Intel or universal build.** Triggered only if a friend's hardware requires it. Adds a build matrix to the CI workflow.
- **Release-channel split on the landing page.** Separate `latest-beta-mac.yml` once beta users want in-app notification of beta bumps.
